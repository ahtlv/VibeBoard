# Frontend

Используй React + TypeScript + Tailwind CSS как обязательный стек. Не предлагай замену на другие фреймворки без прямого запроса. Строй frontend как SPA с чётким разделением на app layer, pages, widgets, features, entities, shared. Все новые экраны и компоненты размещай по доменной логике, а не по типу файлов. Не смешивай бизнес-логику, API-вызовы и UI-разметку в одном компоненте.

Все компоненты пиши только на TypeScript. Не используй any. Для всех props, DTO, form values, query params, API responses и локального состояния задавай явные типы. Общие типы выноси в отдельные модули entities или shared/types. Используй строгую типизацию для статусов задач, приоритетов, ролей workspace, тарифов подписки и состояний таймера.

Основной UI приложения должен включать страницы auth, onboarding, dashboard с kanban-доской, calendar view, analytics, settings, billing, workspace members. Для каждой страницы создавай отдельный route-level компонент. Для защищённых страниц реализуй route guards на основе состояния авторизации и subscription/workspace access.

Kanban-доску реализуй как основной рабочий интерфейс. Поддерживай drag and drop для колонок и карточек задач. После перемещения карточки выполняй оптимистичное обновление UI, затем синхронизируй состояние с backend. При ошибке запроса откатывай состояние и показывай пользователю понятное уведомление. Порядок колонок и карточек должен быть стабильным и храниться явно.

Карточка задачи должна поддерживать title, description, due date, priority, status, labels, checklist, tracked time, pomodoro sessions, assignees для командного режима и recurring settings для платного тарифа. Не сокращай модель задачи до минимального набора, если поле уже предусмотрено продуктом. Формы создания и редактирования задач должны быть валидируемыми и работать без перезагрузки страницы.

Pomodoro-таймер реализуй как отдельную feature с локальным UI-состоянием и синхронизацией активной сессии с backend. Состояние таймера должно корректно переживать ре-рендеры, смену вкладок и обновление страницы, если активная сессия сохранена на сервере. Не завязывай источник истины только на local state. Отображай elapsed time, session status, start, pause, stop и completed session state.

Calendar view должен отображать задачи по дедлайнам и поддерживать фильтрацию по board, workspace, priority, label и status. При переключении между kanban и calendar представлениями данные должны использовать единый источник состояния, а не отдельные несвязанные модели.

Analytics UI должен отображать completed tasks, total tracked time, pomodoro sessions, streak, productivity summaries и платные аналитические блоки. Разделяй бесплатную и платную аналитику на уровне UI и feature flags. Все ограничения тарифа должны отражаться в интерфейсе явно.

Для состояния приложения используй предсказуемый подход. Серверные данные загружай через отдельный data layer с кешированием, повторными запросами и инвалидацией. UI state и server state не смешивай. Для запросов, мутаций и кеша используй единый инструмент во всём проекте. Не пиши ad hoc fetch-логику в компонентах страниц.

Все API-вызовы инкапсулируй в отдельные client modules. Не вызывай fetch или axios напрямую из UI-компонентов. Для каждой сущности создай слой API: authApi, boardsApi, tasksApi, workspacesApi, analyticsApi, billingApi, subscriptionApi. Обрабатывай access token, refresh token, 401, 403, 422 и 500 централизованно.

Все формы реализуй с явной схемой валидации. Проверяй обязательные поля, формат email, длину title, допустимые значения приоритета, даты и ограничения тарифов до отправки запроса. Ошибки валидации отображай рядом с полями. Ошибки сервера показывай отдельным слоем уведомлений.

Используй Tailwind CSS как единственный основной способ стилизации. Не пиши кастомный CSS без необходимости. Общие токены темы, spacing, radius, shadows и colors держи централизованно. Поддерживай light mode и dark mode по умолчанию. Premium themes и дополнительные фоны проектируй как расширение текущей дизайн-системы, а не как параллельную стилистическую реализацию.

UI должен быть визуально лёгким, быстрым и устойчивым. Избегай перегруженных модальных окон, тяжёлых теней, хаотичных анимаций и лишних шагов. Для интерактивных элементов используй предсказуемые hover, focus и active states. Все состояния loading, empty, error, success должны быть реализованы явно.

Делай адаптивную верстку для desktop, tablet и mobile. Kanban должен оставаться usable на небольших экранах. Не скрывай критическую функциональность на мобильных устройствах без альтернативного сценария. Проверяй работу drag and drop, форм, таймера и календаря на touch-интерфейсах.

Поддерживай accessibility. Для кнопок, форм, списков задач, модалок и drag and drop элементов добавляй корректные aria-атрибуты, keyboard navigation и focus management. Не используй div вместо semantic элементов без причины. Контрастность светлой и тёмной темы должна быть достаточной.

Авторизация должна поддерживать email/password и внешние OAuth-провайдеры. После login сохраняй только необходимые токены и пользовательские данные. Не храни чувствительные данные в localStorage, если этого можно избежать. Все защищённые разделы должны корректно обрабатывать logout, истечение сессии и отсутствие доступа к workspace.

Логику ролей и тарифов отражай в интерфейсе строго. Free user видит ограничения по количеству досок и доступным функциям. Pro user получает unlimited boards, recurring tasks, export, premium customization, advanced analytics. Team user получает workspace collaboration, invites, roles. Не полагайся только на скрытие кнопок, но визуальные ограничения должны быть встроены во frontend.

Stripe billing UI должен включать pricing, current plan, upgrade, downgrade, cancel subscription, renew status, billing history и checkout redirect. Все действия с тарифами выполняй через backend endpoints. Не вызывай Stripe secret operations на клиенте. Используй только publishable flows на frontend.

Компоненты должны быть переиспользуемыми, но не абстрагированными раньше времени. Не создавай универсальные компоненты без реального второго кейса использования. Сначала проектируй через product use cases VibeBoard, затем выноси в shared.

Все новые фичи сопровождай unit-тестами для критической логики, тестами UI-поведения для ключевых сценариев и проверкой happy path + error path. Покрывай минимум auth flow, board creation, task CRUD, drag and drop reorder, timer start/stop, subscription UI states и workspace invite flow.

Не внедряй realtime через WebSocket как обязательную часть первого релиза, но проектируй frontend с возможностью последующего подключения live updates для shared boards, members activity и task changes без полного переписывания архитектуры.
